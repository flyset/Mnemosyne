# TRACK 007 [COMPLETED]: consent-gated memory remember

Track
- ID: TRACK_007
- Repository: Mnemosyne
- Branch: main
- Current path: .backlog/COMPLETED/2026/TRACK_007_COMPLETED_consent_gated_memory_remember.md

Problems (PORE)
- P1: As a user, I cannot ask an MCP-compatible agent to save an approved memory, because Mnemosyne exposes recall but no durable creation Tool.
- P2: As an operator, I cannot safely enable agent-initiated memory creation, because mutation is disabled inside the shared service and there is no server-side Tool gate paired with an MCP-client per-call approval rule.
- P3: As a Tool caller, I cannot reliably distinguish creation, active duplicate, archived duplicate, validation refusal, policy refusal, conflict, disabled, and storage-failure outcomes, because no public `memory_remember` contract exists.
- P4: As a user governing durable context, I cannot verify that remember calls avoid secrets, uncontrolled paths, hidden operational fields, and content-bearing logs, because those public mutation boundaries have not been specified or tested.

Objective
- Expose one narrow, disabled-by-default `memory_remember` MCP Tool that persists an explicitly approved canonical version-2 record through the shared memory service only when the operator enables the Tool and the MCP client asks for approval on every exact call.

Non-negotiables
- This Track remains planning-only while DRAFT; no implementation or implementation-driving tests begin until all blocking questions are resolved and the Track is moved to ACTIVE.
- All implementation follows TDD: a focused failing test, the smallest passing implementation, then refactoring and validation.
- `mnemosyne/memory/` remains the owner of memory meaning, validation, duplicate policy, atomic persistence, and mutation-disabled lifecycle behavior; the MCP package is a thin adapter.
- `memory_remember` is disabled and absent from Tool discovery by default. Enabling the MCP server itself does not enable memory mutation.
- Durable mutation requires the MCP client's approval of the exact Tool call. No `confirmed`, `approved`, `consent`, or equivalent model-supplied argument is accepted as evidence of consent.
- Clients that cannot require per-call approval must not be used with the mutation Tool enabled. In OpenCode, users must choose `once` for every call; session-wide `always` approval and auto-approval both defeat this boundary and are unsupported while mutation is enabled.
- The Tool accepts no filesystem path, record ID, timestamp, lifecycle state, revision, or `recorded_via` value from the caller; those operational fields remain server-controlled.
- Memory remains local-first, single-user, filesystem-backed, directly visible, and easy to delete.
- The Tool must not store secrets, tokens, private keys, credentials, or disallowed sensitive personal data. The enforceable refusal boundary must be resolved before activation.
- Validation, errors, results, and logs must not disclose absolute paths or unrelated records. Logs must not include title, content, tags, complete arguments, or rejected secret material.
- Automated tests use temporary roots only. Direct MCP checks may write only approved temporary records and must remove all resulting data.
- No commit or push occurs unless explicitly requested.

Why now / impact
- Track 006 established and tested the canonical shared record, path, store, duplicate, and lifecycle service contracts. Exposing remember now can reuse those foundations without duplicating domain logic, and it is the smallest step that turns Mnemosyne from externally read-only recall into user-governed durable memory.
- Defining operator enablement, client approval, refusal policy, and observable outcomes before the first public mutation prevents accidental writes and incompatible durable records from becoming part of the contract.

Risks / decisions
- Risk: Merely exposing a mutation Tool can permit automatic durable writes when a client defaults unknown Tools to allow or runs in auto-approval mode.
- Risk: A model-provided confirmation field creates the appearance of consent without an enforceable user boundary.
- Risk: A broad or hand-maintained Tool schema can drift from shared scope, namespace, kind, and normalization rules.
- Risk: Returning or logging complete records can unnecessarily duplicate sensitive memory content outside the visible source-of-truth file.
- Risk: Naive secret detection can both miss dangerous values and reject harmless memory; the enforceable policy and its limitations must be explicit.
- Risk: Dynamic Tool availability may be cached by a running MCP client, requiring server and client restart/reconnection after enablement changes.
- Risk: Duplicate discovery remains bounded by the shared store candidate limit and can refuse creation rather than risk an unchecked duplicate.
- Confirmed risk: In the installed OpenCode 1.17.11 merged agent permission order, exact `mnemosyne_memory_remember: ask` appeared before the externally configured broad `mnemosyne_*: allow`; the first configured-client call executed without a prompt. Schema/debug-config presence alone therefore does not prove effective per-call approval.
- Decision: This Track exposes remember only; lifecycle mutation Tools require later independent Tracks and approval contracts.
- Decision: The existing shared domain and filesystem files remain the only source of memory truth; the MCP adapter introduces no second schema or index.
- Decision: The exact proposed memory is visible in Tool arguments before client approval, while the Tool result and server logs remain minimized.
- Decision: Server-generated IDs, timestamps, lifecycle, revision, and `recorded_via` remain outside the public input schema.
- Decision: Default-off means absent from both discovery and dispatch, not merely a handler that advertises itself and then returns `mutation_disabled`.
- Decision: Automated and direct protocol validation use isolated temporary roots and physically remove resulting files.

Decision log
- Decision (prior Track 006): Durable writes require per-call MCP-client approval after the exact proposed memory is visible; denial produces no server call, and a model-provided confirmation field is never consent.
- Decision (prior Track 006): Mutation Tools are disabled by default and clients that cannot enforce per-call approval must leave them disabled.
- Decision (prior Track 006): `MemoryService.remember()` owns mutation enablement, duplicate detection, server-generated operational fields, and canonical persistence; MCP handlers adapt but do not redefine those behaviors.
- Decision (prior Track 006): Exact active duplicates return `already_exists`; exact archived duplicates return `existing_archived`; neither creates another record.
- Decision (prior Track 006): Files are the only durable source of truth, writes are private and atomic, and mutation logs omit title, content, tags, deleted data, absolute paths, and complete arguments.
- Decision (Track 007 draft): OpenCode's current published schema permits arbitrary Tool-name permission keys with `ask`, and MCP Tools are prefixed by server name; Q8 records the verified exact key, merged configuration, and runtime limitations.
- Decision (Track 007 draft): OpenCode configuration changes are loaded only at startup, and `always`, `--auto`, or interactive auto-approval can bypass repeated `ask` prompts; enablement documentation and acceptance checks must make those limitations explicit.
- Decision (Q1): `memory_remember` accepts exactly the nine caller-owned `MemoryDraft` fields: `scope`, `namespace`, `collection`, `kind`, `language`, `title`, `content`, `tags`, and `origin`. All nine keys are required and unknown keys are rejected at every object level. `namespace` always contains required `kind`, `id`, and nullable `label`; `collection` is either null or an object with required `id` and nullable `label`; `language` and `title` are required but nullable; `tags` is a required zero-to-ten-item array. IDs use the canonical 1-64-character identifier pattern; labels are 1-100 non-whitespace characters when present, language is null or a canonical-compatible 1-35-character language tag, title is null or 1-200 non-whitespace characters, content is 1-4,000 non-whitespace characters, and each unique tag is 1-50 non-whitespace characters. The schema uses one scope branch per `SCOPE_DEFINITIONS` entry and derives each allowed namespace kind and memory kind from the shared domain. Public `origin` permits only `explicit_user_statement` and `user_approved_proposal`; server/import-only origins remain outside the Tool contract.
- Decision (Q2): The only remember gate is `MNEMOSYNE_MEMORY_REMEMBER_ENABLED`. An absent value and the exact lowercase value `false` disable remember; the exact lowercase value `true` enables it. Empty, padded, differently cased, numeric, or any other values raise a bounded configuration `ValueError` without echoing the supplied value and fail server startup closed. The value is parsed once while constructing the startup registry, enables only `memory_remember`, and never enables later mutation Tools.
- Decision (Q3): `registry.py` will expose a deterministic registry value built by a side-effect-free `build_tool_registry(memory_remember_enabled: bool)` seam. The startup registry parses the operator setting once, preserves the existing ordered default definitions and handlers, conditionally appends remember definition/dispatch together, and supplies that same immutable selection to both MCP `tools/list` and the `list_tools` handler. Focused tests call the builder directly for enabled and disabled states; runtime code does not mutate a global registry after startup.
- Decision (Q4): Remembered and duplicate calls return a normal Tool result whose JSON text is `status` plus `reference` and `lifecycle`. `reference` contains exactly `scope`, `namespace_id`, nullable `collection_id`, and `id`; `lifecycle` contains exactly `state` and positive `revision`. Status is one of `remembered`, `already_exists`, or `existing_archived`. Results omit path, namespace/collection labels, kind, language, title, content, tags, provenance, timestamps, and internal scores.
- Decision (Q5): `remembered`, `already_exists`, and `existing_archived` are the only ordinary non-error outcomes. Every refusal or failure has `isError: true` and bounded JSON text. `MemoryValidationError` maps to `status: invalid_request`, its controlled `field`, and these fixed public code/messages: `invalid_scope` / `scope is invalid`; `invalid_namespace` / `namespace is invalid for scope`; `invalid_collection` / `collection is invalid`; `invalid_kind` / `kind is invalid for scope`; or `invalid_record` / `memory field is invalid`. The Tool-owned public-origin restriction maps to `status: invalid_request`, `code: invalid_origin`, `field: origin`, and `message: origin is invalid`. Only controlled field names from the draft contract may be returned; anything else becomes `field: memory`. Shared content-policy refusal maps to `status: refused`, `code: disallowed_content`, and `message: memory contains content that Mnemosyne does not store`. `MutationDisabled` maps to `status: policy_error`, `code: mutation_disabled`, and `message: memory remember is disabled`. `CandidateLimitExceeded` maps to `status: storage_error`, `code: candidate_limit_exceeded`, and `message: memory scope contains more than 1000 candidate files`. `WriteConflict` maps to `status: conflict`, `code: write_conflict`, and `message: generated memory identity conflicts with an existing record`. `UnsafeMemoryPath`, `MemorySourceUnavailable`, and `OSError` map to `status: storage_error`, `code: memory_source_unavailable`, and `message: memory could not be stored`. An unexpected exception maps to `status: internal_error`, `code: internal_error`, and the same `memory could not be stored` message; no exception text reaches the Tool result.
- Decision (Q6): The shared domain will own a pure remember-content policy that runs after canonical draft validation and before duplicate discovery or directory creation. It inspects all caller-owned free-form strings, including namespace/collection IDs and labels, title, content, and tags, but never logs or returns a match. After canonical normalization it rejects: PEM headers for RSA, EC, DSA, OpenSSH, generic, or PGP private keys; AWS `AKIA`/`ASIA` plus 16 uppercase alphanumeric characters; GitHub `ghp_`, `gho_`, `ghu_`, `ghs_`, `ghr_`, or `github_pat_` tokens with at least 20 permitted payload characters; Slack `xoxb-`, `xoxa-`, `xoxp-`, `xoxr-`, or `xoxs-` tokens with at least 10 payload characters; Stripe `sk_live_`/`rk_live_` tokens with at least 16 alphanumeric characters; Google `AIza` plus 35 URL-safe characters; and `sk-` tokens with at least 20 URL-safe characters. It also rejects case-insensitive Basic/Bearer authorization headers; assignments to `password`, `passwd`, `pwd`, `secret`, `api_key`, `apikey`, `access_token`, `refresh_token`, or `client_secret` followed by `:` or `=` and a non-whitespace value of at least four characters; URI user-info containing a non-empty password; compact JWT-shaped values with exactly three non-empty base64url segments separated by periods; digit sequences containing 13-19 digits after spaces/hyphens are removed when they pass Luhn; and standalone `NNN-NN-NNNN` SSN-shaped values. Matching is substring-based and case-sensitive only where the named provider format is case-sensitive. Synthetic positive and near-miss tests lock these signatures. This is deliberately bounded signature detection rather than semantic DLP: documentation must state that false positives and false negatives remain possible, that unrecognized secrets and sensitive personal facts are still prohibited by policy and Tool instructions, and that users must review the exact proposed arguments before approving. No external service, entropy scanner, or content-bearing audit store is introduced.
- Decision (Q7): Logger `mcp.memory_remember` emits one terminal event per handler call. INFO is used for `remembered`, `already_exists`, and `existing_archived`; WARNING for invalid, refused, disabled, candidate, conflict, and storage outcomes; ERROR for unexpected failures. The only permitted fields are fixed event/outcome text; stable error code and field when applicable; validated scope, namespace kind, memory kind, collection-present boolean, and origin; and, after a domain result, generated memory ID, lifecycle state, and revision. Namespace/collection IDs or labels, title, content, tags, language, complete arguments, rejected values, absolute/relative paths, exception messages, and tracebacks are forbidden.
- Decision (Q8): OpenCode registers the Tool as `mnemosyne_memory_remember`. The published schema permits arbitrary Tool permission keys, and `opencode debug config` validates structure but does not prove effective precedence. Exact top-level and agent-level `ask` rules were insufficient when the merged agent rules placed an external broad `mnemosyne_*: allow` afterward; runtime testing auto-executed the call. S6 is reopened to establish and test a merged rule order in which the exact ask is effective. Config changes require quitting and restarting OpenCode. Configured-client checks use an isolated temporary-root server: `reject` must produce no server event or file, and `once` must display the complete exact Tool arguments and produce one request/write. OpenCode's `always` choice approves later calls for the session, and `--auto` or interactive auto-approval approves `ask`; all three defeat mandatory per-call review and are unsupported while mutation is enabled. Their documented behavior is treated as negative safety evidence, not as valid consent. Any session in which `always` or auto-approval was selected must disable mutation and restart before further use. Other per-agent permission overrides remain an operator responsibility.
- Decision (Q9): When disabled, remember is absent from `tools/list`, `list_tools`, and dispatch, with no separate operator diagnostic Tool or endpoint. Toggling the server setting requires a server restart because registry availability is startup-fixed. OpenCode must also be restarted/reconnected to refresh cached Tool discovery. A stale client may continue displaying a removed Tool, but disabled server dispatch still returns unknown Tool; a stale client may omit a newly enabled Tool until reconnection.
- Decision (Q10): Tests will preserve the shared-domain no-MCP/FastAPI/route import boundary; assert that `memory_remember/` contains only `__init__.py`, `definition.py`, and `handler.py`; assert its definition derives scope/dimension branches from shared definitions; and assert its handler imports shared draft, service, store, policy/error, and settings contracts without importing recall internals or route/FastAPI modules. Registry tests independently prove stable ordering, matching discovery/dispatch, and the unchanged default list. Handler/package-export tests prove the adapter adds no second record schema or persistence implementation.
- Decision (Q11): Safe direct validation first runs the default-off server and checks both discovery methods, unknown direct dispatch, and absence of any root creation. A fresh process then runs with exact enablement and an approved temporary memory root, lists the Tool, remembers one benign synthetic project decision, inspects the single canonical file, repeats the exact call to prove duplicate identity and one-file idempotence, recalls it through `memory_recall`, and physically removes the entire temporary root in cleanup. Configured OpenCode checks use the same isolated server/root to prove reject/no-call and `once`/one-call behavior when interactive testing is available. No ad-hoc MCP script or real user memory is created.
